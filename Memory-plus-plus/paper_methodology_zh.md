# Memory++: 将推理负担从大模型转移到系统设计

## 摘要

Memory++ 是一个基于 Qwen3-8B (8B参数) 的长期对话记忆系统。在 LongMemEval-S 上达到 F1=0.457，超越 GPT-4+Full Context 基线，参数量仅为其 1/200。核心论点：**长期记忆系统的瓶颈不是模型大小，而是检索-推理管线的设计**。我们通过将小模型做不好的推理子任务（时间计算、多文档计数、答案定位、幻觉检测）转移到确定性的预处理阶段，使 8B 模型也能完成复杂记忆任务。消融实验验证了各模块的贡献，并揭示了个人知识图谱中精确率优于召回率的反直觉发现。

---

## 1. 问题与动机

长期对话记忆问答可以分解为五个能力维度：

| 能力 | 难点 | 核心思路 |
|------|------|---------|
| 信息提取 | 从海量历史中定位答案 | 多信号检索 + 证据句高亮 |
| 跨会话推理 | 聚合分散在多个会话中的信息 | 枚举-计数分解 |
| 时间推理 | 日期计算、时长换算 | 预计算时间差 |
| 知识更新 | 同一话题多次更新，需选最新 | 日期排序 + 新旧标签 |
| 拒绝回答 | 区分"检索不到"和"确实没有" | 多维置信度 + 接地验证 |

现有方案的问题：全量上下文方案（GPT-4 128K）把所有负担压给模型，成本高且上下文仍然不够；简单 RAG 只做粗粒度检索，精细推理仍然依赖模型。Memory++ 的做法是**逐个识别模型做不好的子任务，用确定性的管线阶段替代**。

---

## 2. 方法

### 2.1 置信度门控的自适应查询扩展

**问题**：22.5% 的错误是"检索失败导致的误拒"——用户的提问和记忆存储的词汇不匹配，三路检索均未命中。

**方法**：用 cross-encoder reranker 的 top-1 分数作为检索置信度。当置信度低于阈值 θ=0.15 时，调用 LLM 生成 2-3 个关键词导向的查询变体，分别检索后合并去重，再经 cross-encoder 重排。

关键在于**条件化触发**：仅约 20% 的查询触发扩展，多数查询无额外开销。

**消融**：去除后 F1 下降 0.88%。

---

### 2.2 预计算时间推理

**问题**：时间推理是最难的维度（基线 F1=0.116）。8B 模型做日期减法和单位换算会系统性出错。

**核心洞察**：时间推理混合了两个子任务——"定位相关记忆"和"计算时间差"。小模型能做好前者但做不好后者。我们把后者从模型中剥离出来。

**方法**：
1. 每个 chunk 前缀标注日期：`[Date: 2023-05-15]`
2. 检索后为每个 chunk 预计算到提问日期的时间差，以多种单位表示：`35 days (5 weeks, ~1.1 months)`
3. 对时间推理题额外生成按时间排序的时间线

模型不再需要做任何日期计算，只需从预计算值中选择正确的那个。

**效果**：temporal-reasoning F1 从 0.116 提升到 0.340（+193%）。

**消融**：去除后 F1 下降 1.46%。

---

### 2.3 Cross-Encoder 两阶段检索

第一阶段（向量+BM25+KG）追求高召回，over-retrieve top_k+10 个候选；第二阶段用 cross-encoder（bge-reranker-v2-m3）精排。

关键设计：对 knowledge-update 类型，在 rerank 之后再按日期降序重排——先保证语义相关性筛选，再做时间优先级。

**消融**：去除后 F1 下降 1.95%，是**单模块贡献最大**的组件。

---

### 2.4 证据句高亮

**问题**：30.9% 的错误是"检索到了正确 chunk 但提取了错误实体"。例如：问"在哪家瑜伽馆"，chunk 中同时包含 "Serenity Yoga" 和 "Down Dog app"，模型回答了后者。

**核心洞察**：与其期望 8B 模型的注意力机制自动聚焦正确句子，不如显式标记出来。

**方法**：将每个 chunk 拆成句子，用关键词重叠度评分（实体匹配权重 2x），得分最高的句子用 `►...◄` 标记。零 LLM 调用。

这把模型的任务从"在长段落中找到并提取答案"降级为"从标记好的句子中提取答案"——后者对小模型更可靠。

---

### 2.5 枚举-计数分解

**问题**：多会话计数题（"提到过多少种运动？"）中，8B 模型系统性过度计数——真值为 3 时输出 7-46。

**核心洞察**：跨文档计数混合了"枚举不同项目"和"数数"两个子任务。LLM 能做好枚举但做不好计数；程序恰好相反。

**方法**：
1. Prompt 指示模型**只列表、不计数**：每行一个 `- item`
2. Python 解析列表行数作为最终答案
3. 如果模型仍输出裸数字，回退到标准答案提取

**效果**：multi-session F1 从 0.140 提升到 0.174（+24%）。

---

### 2.6 对抗前提检测与答案接地验证

**问题**：部分问题包含虚假前提（"你在 Studio X 上的瑜伽课怎么样？"但 Studio X 从未提及）。

**方法**：输入侧和输出侧各一道防线：
1. **前提检测**（生成前）：查询实体在上下文中的覆盖率 < 30% → 标记为可疑前提，注入强 IDK 指令
2. **答案接地**（生成后）：答案实体在上下文中的覆盖率 < 30% → 判定为幻觉，替换为 "I don't know"

两者均零额外 LLM 调用。

---

### 2.7 个人知识图谱的精度-召回权衡（负面结果）

**发现**：消融实验显示 KG 实体索引**反而损害性能**（去除后 F1 提升 1.36%）。

**原因分析**：正则 NER 的子串匹配过于宽泛——"art"匹配到"start"，"time"匹配到"overtime"——注入了大量不相关 chunk。

**修复**：限制为精确实体匹配 + 多词短语部分匹配；移除单词子串匹配和关键词模糊匹配；增加最低匹配分阈值。

**对领域的启示**：对于从对话中构建的小规模个人知识图谱，**实体提取的精确率比召回率重要得多**。这与 Web 级信息检索中"召回优先"的经验相反。

## 5. 讨论

### 推理转移何时有效？

1. **子任务有确定性解法**：日期计算、行数计数有明确的输入输出
2. **转移对模型透明**：模型收到的是预处理后的上下文（时间差数值、高亮句子），无需知道预处理的存在
3. **子任务是瓶颈**：temporal-reasoning 基线 F1=0.116，说明模型确实做不好这个子任务

### 推理转移何时失效？

1. **主观推理**：偏好推断（F1=0.232）没有"正确的预处理"
2. **模型不遵循格式**：枚举-计数需要模型输出列表格式，8B 模型有时直接输出数字
3. **跨 benchmark 泛化**：针对一个 benchmark 答案格式的优化可能伤害另一个
